# ADR-0003: Capturing Global Channels in Dynamic Templates via Binding and Key Domains

- **Status**: Proposed  
- **Date**: 2026-03-26  
- **Depends on**: ADR-0002, ADR-0001  
- **Decision scope**: Разрешение использования глобальных каналов в dynamic templates через механизм захвата и связывания; устранение конфликтов потребления через домены ключей (static vs dynamic) посредством канонического кодирования ключа.

---

## Context

ADR-0002 вводит жёсткое разделение static rules и dynamic templates, запрещая глобальные каналы в динамических паттернах. Это упрощает модель, но ограничивает повторное использование общих каналов в локальных компонентах.

Если разрешить dynamic templates потреблять токены из глобального канала `G` (т.е. из очереди `Q[G.channel_id][key]`), возникает риск конкуренции между:
- static rules, которые также потребляют `G(key)`,
- dynamic инстансами, которые потребляют `G(key)` через связанное представление.

Для сохранения композиционности требуется гарантировать, что токены, предназначенные для dynamic инстансов, не конкурируют с токенами, предназначенными для static rules, даже если номинально используются один и тот же `channel_id`.

---

## Decision

### D1. Ослабление запрета ADR-0002: разрешить захват глобальных каналов
Dynamic templates могут использовать глобальные каналы в паттернах только через механизм **capture + bind**, реализуемый компилятором/рантаймом. Пользователь не управляет ключом связывания явно.

### D2. Операция захвата и связывания (capture/bind)
Вводится операция:

- `capture(G)` внутри dynamic template, которая создаёт bound handle `g_bound` для `G` в контексте данного инстанса.

Семантика:
- `g_bound` является bound-представлением глобального канала `G`;
- key связывания не задаётся пользователем и определяется рантаймом как `instance_id` данного инстанса.

Требование:
- В dynamic templates запрещено произвольное связывание по пользовательскому ключу; разрешено только связывание к собственному `instance_id`.

### D3. Домены ключей: StaticDomain и DynamicDomain
Вводятся два непересекающихся домена ключей:

- **StaticDomain**: ключи, используемые static rules (произвольные бизнес-ключи).
- **DynamicDomain**: ключи, используемые dynamic инстансами (`instance_id`).

Разделение доменов реализуется каноническим кодированием Kafka message key:

- `EncodedKey = Encode(domain, raw_key)`

где:
- для static сообщений используется `Encode(StaticDomain, raw_business_key)`;
- для dynamic сообщений используется `Encode(DynamicDomain, instance_id)`.

В простейшей форме Encode может быть реализована как добавление внутреннего префикса (`stat_`, `dyn_`), однако для корректности и исключения коллизий Encode является внутренней функцией рантайма, а не пользовательской строковой конкатенацией. Рантайм обязан:
- запрещать пользовательские `raw_business_key`, которые могут приводить к пересечению доменов;
- обеспечивать взаимно-однозначное декодирование (при необходимости).

### D4. Применение доменов ключей к matching
Static rules:
- потребляют только сообщения с `EncodedKey` из `StaticDomain`.

Dynamic templates:
- потребляют только сообщения с `EncodedKey` из `DynamicDomain`.

Захваченный глобальный канал `g_bound` в dynamic template потребляет сообщения:
- с тем же `channel_id`, что и `G`,
- но только в `DynamicDomain` (т.е. key = `Encode(DynamicDomain, instance_id)`).

Это исключает конкуренцию static и dynamic подсистем за одну и ту же очередь `Q[channel_id][key]`, поскольку `key` всегда различается по домену.

### D5. Правила паттернов в dynamic templates (обновление ADR-0002 D4/D5)
Dynamic templates могут использовать в паттернах:
- bound каналы инстанса;
- bound handle’ы, полученные через `capture()` глобальных каналов.

Прямое использование unbound глобального канала в паттерне остаётся запрещённым.

## Consequences

### Positive
- Разрешается повторное использование глобальных каналов внутри dynamic templates без явной работы пользователя с ключами.
- Исключаются ownership-конфликты между static rules и dynamic templates за счёт строгого разделения key-space.
- Сохраняется композиционность: добавление динамического инстанса не изменяет поведение статических правил в StaticDomain.

### Negative / Trade-offs
- Усложняется модель ключей и формат Kafka message key (вводится доменное кодирование).
- Появляется дополнительная внутренняя операция `capture()` и правило, что связывание возможно только с `instance_id`.
- Непосредственный join между static и dynamic доменами невозможен без явного “моста” (не рассматривается в данной ADR).

## Open questions / Next steps
- Точное определение `Encode(domain, raw_key)` (строковый префикс vs бинарный тег), требования к каноничности и обратимости.
- Инструменты диагностики: отображение `EncodedKey` в человекочитаемый вид в логах/трейсах.
- Модель взаимодействия между доменами (если потребуется): явные bridge-компоненты, семантика доставки и влияние на EOS.
