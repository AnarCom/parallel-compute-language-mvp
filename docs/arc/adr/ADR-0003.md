# ADR-0003: Захват глобальных каналов в dynamic rule и домены ключей (stat_/dyn_)

- **Status**: Proposed  
- **Date**: 2026-03-26  
- **Depends on**: ADR-0002, ADR-0001  
- **Decision scope**: Разрешение использования глобальных каналов в паттернах dynamic rule через механизм захвата и связывания; введение доменов ключей `stat_` и `dyn_` для устранения конфликтов между static rule и dynamic rule.

---

## Context

ADR-0002 вводит жёсткое разделение: global channel не может участвовать в паттерне dynamic rule. Это устраняет необходимость распространения правил и упрощает модель, но ограничивает повторное использование глобальных каналов внутри инкапсулированных компонентов.

Если разрешить, чтобы global channel участвовал в паттерне dynamic rule, то необходимо решить две задачи:

1) **Локализация исполнения**: dynamic rule должен оставаться исполнимым на воркере, выбранном по ключу экземпляра (без распространения правил).
2) **Разделение потоков токенов**: токены, предназначенные для static rule, не должны конкурировать с токенами, предназначенными для dynamic rule, даже если используется один и тот же `channel_id`.

Для решения (2) вводится разделение пространств ключей на два домена: `stat_` для static rule и `dyn_` для dynamic rule.

---

## Definitions

- **raw key** — ключ, указанный пользователем для unbound каналов (бизнес-ключ).
- **instance_id** — UUID ключ экземпляра dynamic rule (как в ADR-0002).
- **encoded key** — ключ Kafka-сообщения после внутреннего кодирования доменом.

Домены ключей:
- `stat_` — домен ключей для static rule;
- `dyn_` — домен ключей для dynamic rule.

---

## Decision

### D1. Механизм захвата и связывания глобального канала
Вводится операция **захвата** глобального канала внутри dynamic rule:

- При создании экземпляра dynamic rule любой глобальный канал, упомянутый в паттерне, автоматически используется как **bound** канал, связанный с `instance_id` данного экземпляра.

Пользователь не имеет синтаксиса для явного связывания ключей и не может выбрать ключ связывания. Связывание выполняется только с `instance_id`.

### D2. Домены ключей `stat_` и `dyn_`
Для исключения пересечения и конкуренции токенов вводится внутреннее кодирование ключей:

- Для токенов, участвующих в matching static rule, Kafka message key = `EncodeKey(stat_, raw_key)`.
- Для токенов, участвующих в matching dynamic rule (включая захваченные глобальные каналы), Kafka message key = `EncodeKey(dyn_, instance_id)`.

`EncodeKey` является внутренней функцией рантайма. Пользователь не формирует префиксы вручную.

### D3. Область применимости доменов
- Static rules обрабатывают только токены с ключом в домене `stat_`.
- Dynamic rules обрабатывают только токены с ключом в домене `dyn_`.

Следствие: даже при использовании одинакового `channel_id` static rule и dynamic rule работают с непересекающимися ключами Kafka и, соответственно, с непересекающимися очередями токенов.

### D4. Обновление правил ADR-0002 (ослабление запрета)
В ADR-0002 D5 (“запрет глобальных каналов в dynamic rule”) заменяется на:

- Глобальные каналы **разрешены** в паттернах dynamic rule только в режиме автоматического захвата и связывания с `instance_id`, и только в домене ключей `dyn_`.
- Прямое использование глобального канала как unbound канала в паттерне dynamic rule остаётся запрещённым.

---

## Consequences

### Positive
- Dynamic rule может использовать глобальные каналы в паттернах без распространения правил по кластеру: ключ `instance_id` локализует исполнение.
- Токены static rule и dynamic rule не пересекаются за счёт доменов ключей `stat_` и `dyn_`.
- Пользователь не управляет связыванием ключей вручную; связывание выполняется детерминированно и единообразно рантаймом.

### Negative / Trade-offs
- Усложняется внутреннее кодирование ключей (появляется `EncodeKey` и домены).
- Прямое взаимодействие между static rule и dynamic rule через один и тот же raw key невозможно без дополнительных механизмов преобразования доменов (не рассматривается в данной ADR).

---

## Open questions / Next steps
- Точное определение `EncodeKey`: строковый префикс `stat_`/`dyn_` или бинарный тег домена; требования к каноничности и исключению коллизий.
- Компиляторные проверки: запрет “ручного” формирования доменов пользователем (если ключ строковый) и запрет использования unbound глобального канала в паттернах dynamic rule.
