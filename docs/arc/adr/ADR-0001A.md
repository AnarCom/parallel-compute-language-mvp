# ADR-0001A: Redis-backed Join Runtime (single-node Redis, current state + queues)

- **Status**: Proposed  
- **Date**: 2026-03-26  
- **Alternative to**: ADR-0001 (Kafka-backed runtime)  
- **Decision scope**: Архитектура runtime при использовании одиночного Redis вместо Kafka; хранение токенов и join-состояния; гарантии атомарности публикации выходных токенов и сохранности входных токенов при сбоях воркеров; модель исполнения и восстановление.

---

## Context

В ADR-0001 токены хранятся в Kafka, а воркеры используют in-memory join-состояние и транзакционную публикацию выходных токенов в Kafka.

В качестве альтернативы рассматривается подход для MVP, где:
- используется **одиночный Redis**;
- система ориентирована на небольшие объёмы (сотни/тысячи ключей и каналов);
- требуется хранить **текущее состояние и очереди**, без требований на долговременный лог и реплей;
- требуется обеспечить минимально необходимые гарантии корректности при сбоях воркеров: отсутствие частичной публикации выходных токенов и отсутствие потери входных токенов при падении воркера.

---

## Decision

### D1. Redis является источником очередей токенов и join-состояния
Redis используется как единое хранилище:
- очередей токенов каналов;
- служебных структур, необходимых для matching, резервирования входов и фиксации выполнения реакций.

Воркеры не используют Kafka для токенов и не опираются на реплей лога для восстановления join-состояния.

### D2. Модель исполнения: воркеры выполняют реакции, Redis обеспечивает атомарность состояния
Реакции исполняются на воркерах. Redis используется для атомарных переходов состояния системы:

- резервирование входных токенов для реакции;
- фиксация выполнения реакции;
- атомарная публикация всех выходных токенов.

### D3. Протокол резервирования и фиксации (reserve/commit)
Для предотвращения потери входных токенов и частичной публикации выходных токенов вводится двухфазный протокол:

1) **reserve**: атомарно выбрать применимое правило, извлечь требуемые токены из очередей каналов и поместить их в резерв реакции, сформировав запись “ожидающей фиксации” реакции.
2) **commit**: атомарно опубликовать все выходные токены и завершить реакцию (снять резерв, пометить выполненной).

Обе фазы выполняются как атомарные операции на Redis (Lua-скрипты или MULTI/EXEC).

### D4. Идентификатор исполнения реакции и идемпотентный commit
Каждая реакция получает `reactionExecutionId` (детерминированный идентификатор), который используется для:

- предотвращения повторной фиксации (idempotent commit);
- безопасного подхвата “ожидающих” реакций другими воркерами при сбоях.

Commit является идемпотентным: повторный commit с тем же `reactionExecutionId` не должен приводить к повторной публикации выходных токенов.

### D5. Подхват незавершённых реакций
Если воркер завершился после reserve и до commit, в Redis остаётся запись “ожидающей” реакции и резерв входных токенов.

Другой воркер должен иметь возможность:
- обнаружить такие реакции (по таймауту/lease);
- повторно исполнить обработчик и выполнить commit (идемпотентно).

### D6. Join-per-key в MVP (рекомендуемое ограничение)
Для упрощения matching и структуры данных применяется ограничение join-per-key:

- matching и резервирование выполняются в пределах одного ключа;
- очереди токенов индексируются по `(channel_id, key)`.

Примечание: при одиночном Redis join-per-key не является обязательным для корректности, но является предпочтительным для эффективности и упрощения реализации.

## Data model (logical)

### Очереди токенов
Для каждой пары `(channel_id, key)` хранится FIFO-очередь токенов:
- `queue:{channel_id}:{key}`

### Резерв входных токенов для реакции
Для `reactionExecutionId` хранится набор зарезервированных входных токенов:
- `reserved:{reactionExecutionId}`

### Ожидающие реакции
Для `reactionExecutionId` хранится запись ожидания (правило, key, lease/таймштамп):
- `pending:{reactionExecutionId}`

### Фиксация выполнения
Флаг/запись, что реакция зафиксирована:
- `done:{reactionExecutionId}`

(Формат ключей и структур является частью реализации и может быть уточнён; в ADR фиксируется логическая необходимость таких сущностей.)

---

## Guarantees (scope)

Гарантии распространяются на публикацию токенов внутри Redis-управляемого протокола reserve/commit.

### G1. Отсутствие частичной публикации выходных токенов
Выходные токены для реакции публикуются атомарно (все либо ни одного) в фазе commit.

### G2. Отсутствие потери входных токенов при сбое воркера
Входные токены не удаляются безвозвратно до commit. До commit они находятся в резерве и могут быть обработаны повторно при подхвате pending реакции.

### G3. Отсутствие повторной публикации выходных токенов при повторном commit
Commit идемпотентен по `reactionExecutionId`: повторное выполнение commit не приводит к повторной публикации выходных токенов.

### G4. Внешние побочные эффекты
Exactly-once семантика для внешних побочных эффектов (HTTP/DB) не обеспечивается. При повторном исполнении реакции прикладной код обязан обеспечивать корректность внешних эффектов (например, через idempotency key на базе `reactionExecutionId`).

## Consequences

### Positive
- Упрощается MVP: отсутствуют partitions/consumer groups/offsets и транзакции Kafka.
- Join-состояние централизовано: не требуется восстановление in-memory состояния через реплей.
- Атомарность публикации выходов и сохранность входов обеспечиваются протоколом reserve/commit на Redis.

### Negative / Trade-offs
- Redis (одиночный) является критической зависимостью и ограничивает масштабирование и отказоустойчивость на уровне инфраструктуры.
- Отсутствует встроенная модель долговременного лога и реплея; отладка и анализ по истории событий затруднены.
- Гарантии корректности существенно зависят от корректной реализации протокола reserve/commit и механизма подхвата pending реакций.
- При переходе на Redis Cluster потребуется переработка ключей и протокола для сохранения атомарности (в рамках одного слота).

## Open questions / Next steps
- Выбор конкретных структур Redis (LIST vs STREAM) и формата токенов.
- Политика lease/timeout для pending реакций и механизм их обнаружения без глобальных сканов.
- Ограничения роста очередей и управление памятью (лимиты, TTL, отбрасывание, DLQ).
- Требования к детерминированности обработчиков относительно входных токенов (для воспроизводимости результатов при подхвате).
