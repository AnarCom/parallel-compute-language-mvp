# ADR-0002: Static vs Dynamic Join Rules and Channel Binding

- **Status**: Proposed  
- **Date**: 2026-03-26  
- **Supersedes/Depends on**: ADR-0001  
- **Decision scope**: Модель программирования: разделение статических и динамических правил; понятия свободных (unbound) и связанных (bound) каналов; ограничения на паттерны; запрет использования глобальных каналов в динамических паттернах.

---

## Context

В системе присутствуют:

- **статические join-правила**, определяемые в глобальной области видимости (compile-time), предназначенные для обработки сообщений в общих (глобальных) каналах;
- **динамические join-правила (шаблоны)**, определяемые внутри функций и инстанцируемые в рантайме, предназначенные для инкапсулированных компонентов с приватными каналами.

Распределённое исполнение основано на Kafka и per-key обработке. Для обеспечения масштабируемости и воспроизводимости требуется детерминированность и строгие правила, исключающие конкуренцию независимых join-машин за одни и те же токены.

Основной риск при смешении глобальных и локальных каналов в динамических паттернах — возникновение конкуренции за токены одной и той же пары `(channel_id, key)` между статическими и динамическими правилами, что нарушает композиционность и инкапсуляцию.

---

## Decision

### D1. Классы правил
Вводятся два класса правил:

1) **Static rules**:  
   - объявляются в глобальной области видимости;
   - составляют фиксированный rule set процесса;
   - предназначены для обработки сообщений в глобальных каналах.

2) **Dynamic templates**:  
   - объявляются внутри функций и инстанцируются в рантайме;
   - форма (structure) каждого шаблона статически известна компилятору;
   - каждый инстанс имеет собственный набор каналов (handle’ов), логически принадлежащих этому инстансу.

### D2. Представления каналов
Вводятся два представления каналов:

- **Unbound channel handle** (свободный канал): отправка/получение сообщений требует явного ключа (key).  
- **Bound channel handle** (связанный канал): ключ неявно зашит в handle; отправка/получение всегда осуществляется с этим ключом.

Примечание: bound/unbound — свойства значения handle; это не обязательно отдельные синтаксические конструкции языка.

### D3. Правило для статических правил: только unbound каналы
Static rules могут использовать в паттернах только **unbound** каналы. Сообщения, обрабатываемые static rules, обязаны содержать **явный ключ**, который участвует в matching как корреляционный идентификатор.

### D4. Правило для динамических шаблонов: только bound каналы инстанса
Dynamic templates могут использовать в паттернах только **bound** каналы, принадлежащие конкретному инстансу шаблона.

- Каналы, созданные внутри dynamic template, считаются bound к ключу инстанса.
- Dynamic template не имеет доступа к unbound каналам в паттернах.

### D5. Жёсткое разделение: глобальные каналы запрещены в динамических паттернах
Использование глобальных каналов (unbound handles, объявленных в глобальной области видимости) внутри паттернов dynamic template **запрещено**.

Запрет распространяется на:
- прямое упоминание глобального канала в `from ...` позиции паттерна;
- любые формы “захвата/связывания” глобального канала для динамического шаблона.

Следствие: dynamic templates не могут потреблять токены, производимые статической подсистемой через глобальные каналы. Взаимодействие возможно только через явные интерфейсы, реализованные на уровне приложения (например, передачей handle’ов), но не через совместное потребление одной очереди токенов.

### D6. Инстанцирование dynamic template
Инстанцирование dynamic template создаёт:

- новый instance identifier `instance_id` (уникальный, генерируется рантаймом);
- набор bound handle’ов каналов инстанса, все из которых привязаны к `instance_id`;
- внутреннюю регистрацию инстанса в рантайме (для отладки/инструментации), при этом rule set процесса не изменяется.

---

## Consequences

### Positive
- Исключается конкуренция static rules и dynamic templates за одну и ту же пару `(channel_id, key)`.
- Сохраняется композиционность: создание динамического инстанса не меняет поведение статических правил.
- Упрощается спецификация и реализация: отсутствуют механизмы захвата/связывания глобальных каналов.
- Обеспечивается инкапсуляция приватных каналов dynamic templates.

### Negative / Limitations
- Dynamic templates не могут непосредственно потреблять токены из глобальных каналов, что ограничивает выразительность и повторное использование инфраструктурных каналов.
- Для интеграции динамических инстансов с внешним окружением требуется передача bound handle’ов через явные интерфейсы (capability passing) либо иные прикладные конструкции.

## Open questions / Next steps
- Формализация правил видимости и “утечки” private bound handle’ов (компиляторные проверки).
- Формализация сериализации/передачи bound handle’ов как значений (capability passing) в замкнутом мире.
- Требования к наблюдаемости: идентификация инстанса в трассировках и метриках.
