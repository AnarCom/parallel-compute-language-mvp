# ADR-0002: Статические и динамические правила и каналы

- **Status**: Proposed  
- **Date**: 2026-03-26  
- **Depends on**: ADR-0001  
- **Decision scope**: Семантика объявления `join`-правил; классификация **static rule** и **dynamic rule**; классификация каналов **unbound** и **bound**; правила использования каналов в паттернах; запрет использования глобальных каналов в паттернах dynamic rule.

---

## Context

В ADR-0001 зафиксировано, что:
- токены хранятся и передаются через Kafka;
- обработка выполняется per-key (один ключ — одна Kafka partition, один воркер);
- matching детерминированен и воспроизводим при реплее.

При этом язык допускает объявление `join`-правил внутри функций. Если разрешить, чтобы выполнение функции на одном воркере создавало **новые каналы** и **новые правила**, то возникает системная проблема:

- токены, адресованные созданным каналам, могут попадать на воркер, который обслуживает соответствующую partition ключа;
- следовательно, правила для этих каналов должны либо:
  1) **всегда исполняться на том же воркере**, где они были созданы, либо  
  2) быть **распространены на все воркеры**, чтобы любой воркер мог их матчить.

Вариант (2) требует доставки/согласования набора правил, обновления индексов matching на всех воркерах и определения момента вступления правил в силу. Это значительно усложняет модель, реализацию и эксплуатацию.

Требуется определить такую семантику, при которой:
- правила, создаваемые при выполнении функций, не требуют распространения по кластеру;
- при этом сохраняется возможность инкапсулировать логику и приватные каналы (например, внутреннее состояние счётчика).

---

## Definitions

- **static rule** — правило, определённое в глобальной области видимости; входит в фиксированный набор правил процесса.
- **dynamic rule** — правило, определённое внутри функции; при выполнении функции создаётся экземпляр dynamic rule.
- **unbound канал** — канал, для которого ключ указывается явно при отправке/получении токена.
- **bound канал** — канал, для которого ключ неявно фиксируется при создании экземпляра dynamic rule; все токены этого канала имеют один и тот же ключ экземпляра.

---

## Decision

### D1. Static rule: только глобальная область видимости
Static rules объявляются только в глобальной области видимости. Набор static rules фиксирован для бинарника и одинаков для всех воркеров, запущенных из данного бинарника.

### D2. Dynamic rule: экземпляры создаются при выполнении функции
Dynamic rules могут объявляться внутри функций. Выполнение функции создаёт экземпляр dynamic rule с собственными bound каналами. Создание экземпляра dynamic rule **не изменяет** набор static rules.

### D3. Привязка экземпляра dynamic rule к ключу (bound)
Каждый экземпляр dynamic rule получает уникальный ключ экземпляра `instance_id` (UUID), формируемый рантаймом. Все bound каналы, созданные в рамках данного экземпляра dynamic rule, используют один и тот же ключ `instance_id`.  

Следствие: все токены bound каналов экземпляра попадают в одну partition и обрабатываются одним воркером (в рамках consumer group), без необходимости распространять новые правила на другие воркеры.

### D4. Правила паттернов

#### D4.1. Паттерны static rule
Паттерны static rule могут использовать только **unbound** каналы. Ключ является частью семантики паттерна и должен быть указан явно (в синтаксисе языка).

#### D4.2. Паттерны dynamic rule
Паттерны dynamic rule могут использовать только **bound** каналы, принадлежащие данному экземпляру dynamic rule.

### D5. Жёсткое разделение: запрет глобальных каналов в dynamic rule
Dynamic rule не может использовать в паттерне какой-либо unbound канал, объявленный в глобальной области видимости (глобальный канал).  

Иными словами:
- global channel **не может** участвовать во входной части паттерна dynamic rule;
- механизмы “захвата” и “связывания” глобального канала для dynamic rule отсутствуют в рамках данной ADR.

---

## Consequences

### Positive
- Экземпляры dynamic rule не требуют распространения по кластеру: их обработка локализована по `instance_id`.
- Семантика кластера остаётся простой: воркеры знают только static rules; dynamic rules остаются внутри “своего” ключа.
- Сохраняется инкапсуляция приватных каналов внутри функций без необходимости отдельной инфраструктуры доставки правил.

### Negative / Limitations
- Dynamic rule не может напрямую принимать входные токены из глобальных каналов. Взаимодействие dynamic rule с внешним окружением возможно только через каналы, переданные/возвращённые из функции (bound каналы экземпляра).
