# ADR-0001: Kafka-backed Join Runtime (per-key join-only, transactional token emission)

- **Status**: Proposed  
- **Date**: 2026-03-26  
- **Decision scope**: Архитектура runtime; хранение/транспорт токенов; модель исполнения; обеспечение атомарности публикации выходных токенов и устранение их дублирования при повторах обработки.

---

## Context

Проектируем распределённый runtime для декларативного языка на основе Join Calculus. Система выполняет реакции (join-правила), потребляющие токены из каналов и порождающие новые токены. В качестве транспорта и журнала токенов на первом этапе используется Kafka. Runtime разворачивается в виде кластера воркеров (например, в Kubernetes).

Требования:
1. Хранение и доставка токенов через Kafka.
2. Масштабирование за счёт Kafka partitions и consumer groups.
3. При повторах обработки входных токенов (например, после сбоя воркера) runtime должен предотвращать дублирование *выходных токенов* в Kafka.
4. Публикация набора выходных токенов должна быть атомарной: “все либо ничего”.
5. Реакции могут выполнять внешние побочные эффекты; их идемпотентность не обеспечивается runtime и является ответственностью прикладного кода.

---

## Decision

### D1. Kafka используется как журнал и транспорт токенов
Kafka рассматривается как единственный источник входных токенов и как целевое хранилище выходных токенов. Для MVP используется один общий topic (например, `domain.tokens`), в котором все токены хранятся совместно; принадлежность токена к каналу кодируется через `channel_id` (например, в headers).

### D2. Явно заданный key используется как Kafka message key
При отправке токена прикладной код обязан явно указать `key`. Runtime размещает `key` в Kafka message key. Партиционирование Kafka по `key` используется как базовый механизм локализации обработки связанных токенов.

### D3. Ограничение модели: per-key join-only
Каждая реакция (join-правило) определяется так, что все токены, участвующие в одном срабатывании реакции, имеют один и тот же `key`. Join между различными `key` не допускается. Это ограничение является частью спецификации MVP и должно проверяться компилятором и/или runtime.

### D4. Модель исполнения
Исполнение реализуется по модели “один поток на partition”:

- Воркеры объединяются в consumer group и получают назначенные Kafka partitions.
- Для каждой назначенной partition воркер выполняет обработку последовательно в рамках одного event loop.
- Matching и выполнение реакций выполняются в том же потоке, что исключает необходимость синхронизации доступа к join-состоянию в MVP.

### D5. Атомарная публикация выходных токенов и устранение их дублирования (Kafka→Kafka)
Для обеспечения атомарности и исключения дублирования выходных токенов при повторной обработке входа используется транзакционный паттерн Kafka “consume-transform-produce”:

- Воркеры читают входные токены из Kafka.
- По результатам matching и выполнения реакций формируют набор выходных токенов.
- Публикуют все выходные токены в рамках Kafka transaction.
- Offset’ы входных сообщений коммитятся в рамках той же транзакции (через `sendOffsetsToTransaction`).
- Транзакция коммитится.

Требование к потребителям выходных токенов: использование `read_committed` для исключения наблюдения сообщений из незакоммиченных/abort’нутых транзакций.

### D6. Внешние побочные эффекты не входят в гарантии runtime
Runtime не обеспечивает exactly-once для внешних побочных эффектов (HTTP/DB и т.п.). Для поддержки прикладной идемпотентности runtime предоставляет детерминированный идентификатор исполнения реакции `reactionExecutionId` (например, функция от `ruleId` и идентификаторов потреблённых входных токенов), который прикладной код может использовать как idempotency key во внешних системах.

### D7. Join-состояние хранится в памяти воркера
Join-состояние для matching определяется как локальные FIFO-очереди токенов `queues[channel_id][key]` и хранится в памяти воркера. Это состояние считается производным от Kafka лога и не реплицируется отдельно в MVP. При сбое воркера восстановление осуществляется за счёт повторного чтения Kafka с последнего закоммиченного offset’а.

### D8. Deterministic matching является обязательным свойством MVP
Matching и выбор срабатываний должны быть детерминированными, чтобы повторное выполнение после сбоя приводило к тем же решениям по потреблению токенов и формированию выходных токенов. В MVP фиксируются:
- FIFO-порядок потребления токенов в пределах `(channel_id, key)`;
- фиксированный порядок/приоритет правил;
- фиксированная политика разрешения конкуренции между несколькими применимыми правилами.

## Matching algorithm (MVP)

### Структуры данных
- `queues[channel_id][key] -> FIFO очередь токенов`
- `rulesByChannel[channel_id] -> список правил, содержащих данный канал`
- Для каждого правила `R`: `deps(R)` (набор каналов), `priority(R)` (или порядок объявления), `handler(R)`

### Инкрементальная обработка
При поступлении токена `(ch, key)`:
1. токен добавляется в `queues[ch][key]`;
2. формируется рабочий список правил `worklist := rulesByChannel[ch]`;
3. далее выполняется итеративный процесс:
   - выбирается следующее правило из `worklist` в соответствии с детерминированной политикой;
   - проверяется применимость правила для данного `key`: все `queues[dep][key]` непусты для `dep ∈ deps(R)`;
   - при применимости правило потребляет по одному токену из каждой требуемой очереди (FIFO) и создаёт “firing” (запланированное исполнение);
   - после потребления токенов рабочий список пополняется правилами, зависящими от затронутых каналов;
4. все запланированные firing’и исполняются последовательно в рамках того же event loop;
5. выходные токены накапливаются в буфере микробатча и публикуются транзакционно согласно D5.

## Consequences

### Положительные последствия
- Реализуемость: архитектура не требует распределённого matcher’а и распределённых транзакций для matching.
- Масштабирование: горизонтальное масштабирование достигается через Kafka partitions при сохранении локальности обработки по `key`.
- Атомарность публикации: выходные токены публикуются “все либо ничего”.
- Устранение дублирования выходных токенов при повторах обработки входа в пределах Kafka→Kafka pipeline.
- Детерминизм упрощает воспроизводимость, отладку и реплей после сбоев.

### Негативные последствия и ограничения
- Ограничение выразительности: join между различными `key` запрещён.
- Join-состояние volatile: при сбое воркера промежуточные “ожидающие” токены восстанавливаются только через реплей Kafka.
- Пропускная способность ограничена моделью “один поток на partition”: длительные реакции и внешние вызовы снижают throughput соответствующей partition.
- Exactly-once для внешних побочных эффектов не обеспечивается; корректность внешних эффектов требует прикладной идемпотентности или иных прикладных механизмов.

## Open questions / Next steps

1. Политика ограничения роста очередей и backpressure (bounded queues, TTL, DLQ).
2. Политика обработки конкурирующих правил (включая случаи пересечения/вложения паттернов) и компиляторные проверки.
3. Наблюдаемость и диагностика: метрики очередей и правил, трассировки firing’ов, интерфейс объяснения “почему правило не применимо”.
4. Сценарии rebalance/failover и требования к стабильности `transactional.id` и управлению жизненным циклом продьюсеров.
